Java基础

线程和进程的区别

进程是程序的一次执行，是系统进行资源分配和调度的独立单位，他的作用就是程序能够独立的运行，提高了系统资源的利用率和吞吐率。

但是进程是系统资源分配和调度的基本单位，因此，进程的创建、销毁、切换都会产生大量的时间和空间的开销。

线程是比进程更小的能独立运行的基本单位，他是进程的一个实体，可以减少程序并发运行时的时间和空间开销，使得操作系统具有更好的并发性。

线程基本不拥有系统资源，只有一些运行时必不可少的资源，比如程序计数器、寄存器和栈，而进程则拥有堆和栈。

synchronized原理

synchronized是java提供的原子性内置锁，这种内置的并且使用者看不见的锁被称为监视器锁，

使用synchronized之后，会在编译之后再同步的代码块前后加上monitorenter和monitorexit字节码指令，他本身依赖操作系统底层互斥锁实现，他的作用是要就是实现原子性操作和解决共享变量的内存可见性问题。

执行monitorenter指令时会尝试获取对象锁，如果对象没有锁或者已经获取到锁，锁的计数器+1,此时其他竞争锁的线程则会进入等待队列中。

执行monitorexit指令则会把计数器-1，当计数器为0时，则锁释放，处于等待队列中线程在继续竞争锁。
synchronized是排它锁，当一个线程获得锁之后，其他线程必须等待该线程释放锁之后才能获得锁，而且由于Java中的线程和操作系统原生线程是一一对应的，线程被阻塞或者唤醒时会从用户态切换到内核态，这个转换过程是非常消耗性能的

从内存语义上来说，加锁的过程会清除工作内存中共享变量，再从主内存中读取，而释放锁的过程就是将工作内存中的共享变量写回主内存中。

synchronized实际上有两个队列waitSet和entryList

  1. 当多个线程进入同步代码块时，首先会进入entryList
  2. 当有一个线程获取到monitor锁后，就赋值给当前线程，并且计数器+1
  3. 如果线程调用wait方法，将释放锁，当前线程置为null，计数器-1，同时进去waitSet等待被唤醒，调用notify或者notifyAll之后则又会进入entryList竞争锁
  4. 如果线程执行完毕，同样释放锁，计数器-1，当前线程置为null

锁的优化机制

从jdk1.6之后，synchronized本身也在不断优化锁的机制，优化机制包括自适应锁、自旋锁、锁消除、锁粗化、轻量级锁、偏向锁以及重量级锁

锁的状态从低到高是无锁–>偏向锁–>轻量级锁–>重量级锁，升级的过程就是从低到高

  • 自旋锁：大部分时候，锁被占用的时间是非常短的，共享变量锁定的时间也很短，所以没有必要挂起线程，用户态和内核态来回上下文切换是很影响性能的。自旋的概念就是让线程执行一个忙循环，可以理解为啥也不干，防止线程从用户态切换到内核态
  • 自适应锁：自适应锁就是自适应的自旋锁，自旋的时间不是固定时间，而是由上一次在同一个锁的自旋时间和锁的持有者状态来决定的
  • 锁消除：锁消除就是jvm检测到一些同步代码块，完全不存在数据竞争的场景，也就是不需要加锁，就会进行锁消除
  • 锁粗化：锁粗化指的是有很多操作都是对同一个对象进行加锁，就会把锁的同步范围扩展到整个操作序列之外
  • 偏向锁：当线程访问同步块获取锁时，会在对象头和栈帧中的锁记录里存储偏向锁的线程id，之后这个线程再次进入同步块时都不需要CAS来加锁和解锁了，偏向锁会永远偏向第一个获得锁的线程，如果后续没有其他线程获得过这个锁，持有锁的线程就永远不需要进行同步，反之，当有其他线程竞争偏向锁时，持有偏向锁的线程就会释放偏向锁
  • 轻量级锁：JVM的对象的对象头中包含有一些锁的标志位，代码进入同步块的时候，jvm就会使用cas的方式来获取锁，如果更新成功则会把对象头中的状态位标记为轻量级锁，如果更新失败，当前线程就会尝试自旋来获取锁

一句话概括就是偏向锁是通过对象头中的偏向线程id来对比，甚至都不需要cas，而轻量级锁是通过cas来修改对象头锁记录和自旋来实现，重量级锁则是除了拥有锁的线程，其他全部阻塞

ReentrantLock和synchronized的区别

  • 二者默认都是非公平锁，但是ReentrantLock可设置为公平锁
  • ReentrantLock等待可中断，当持有锁的线程长时间不释放锁的时候，等待中的线程可以放弃等待，转而处理其他的任务
  • ReentrantLock可以绑定多个Condition条件对象

ReentrantLock是基于AQS（AbstractQueuedSynchronizer抽象队列同步器）实现的

AQS内部会维护一个state状态位，尝试加锁的时候会通过CAS来修改值，如果成功就设置为1 ，并且把当前线程id赋值，则代表加锁成功，一旦获取到锁，其他线程就会被阻塞进入阻塞队列中自旋，获取锁的线程释放锁的时候会唤醒阻塞队列中的线程，释放锁的时候则会把state设置为0，同时把线程id清空

CAS原理

CAS叫做CompareAndSwap,比较并交换，主要是通过处理器的指令来保证操作的原子性，它包含三个操作：

  1. 变量内存地址，V表示
  2. 旧的预期值，A表示
  3. 准备设置的新值，B表示

当执行CAS指令时，只有当v等于a的时候，才会用b去更新v的值，否则就不会执行更新操作

CAS的缺点

  • ABA问题：在cas更新的过程中，先读取到的值了a,然后进行操作，a被改为b了，接着又将b改为a,这时候准备赋值的时候去读取发现还是a，但是实际上值已经经过了a-b-a的过程了
  • 循环时间长开销大：自旋cas的方式如果长时间不成功，会给CPU带来很大的开销
  • 只能保证一个共享变量的原子性：只对一个共享变量操作可以保证原子性，但是多个不行，多个可以通过AtomicReference来处理或者使用synchronized来实现

HashMap原理（简述版）

HashMap主要是由数组和链表组成的，是线程不安全的，1.7和1.8的区别在于，1.7采用头插，1.8采用尾插，采用头插会导致链表死循环，并且1.8加入红黑树之后对性能有提升

put

往map中添加元素的时候首先需要通过对key hash然后与数组长度-1进行与运算((n-1) & hash)，都是2的次幂所以等同于取模，但是位运算的效率更高。找到数组中的位置之后，如果数组中没有元素直接存入，如果存在元素，就判断key是否相同，如果key相同就覆盖，如果不同，就插入到链表的尾部，如果链表长度大于8，则会转成红黑树，最后判断数组长度是否超过默认长度（16）*负载因子（0.75），如果超过就扩容

ConcurrentHashMap原理

ConcurrentHashMap在1.7和1.8改动较大，1.7使用的segmet+hashEntry分段锁来实现的，1.8则是通过Cas+Synchronized+node来实现的，同时也加入了红黑树，避免链表过长导致的性能问题

1.7分段锁

从结构上说，1.7的ConcurrentHashMap采用分段锁机制，里面包含一个Segment数组，Segment继承于ReentrantLock，Segment则包含HashEntry的数组，HashEntry本身就是一个链表，具有kv的能力，同时包含一个能指向下一个节点的指针

实际上就是相当于每个Segment都是一个HashMap,默认Segment的长度是16，也就是说支持最多16个线程的并发写，每个Segment之间互不影响

put

其实可以发现整个流程是和hashmap差不多的，只不过是需要先定位到具体的segment，然后通过ReentrantLock去操作而已

  1. 计算hash，定位到segment，segment如果为空就先初始化
  2. 使用ReentrantLock加锁，如果加锁失败就自旋，自旋超过次数就阻塞等待，保证一定能获取锁成功
  3. 遍历HashEntry,就是和HashMap的操作一样。

get

key通过hash定位到segment，在遍历链表定位到具体的元素，需要注意的是value是volatile修饰的，所以，get是不需要加锁的

1.8 CAS+Synchronized

1.8抛弃了分段锁，转而使用cas+synchronized来实现，同样hashEntry改为node,同时也加入了红黑树

put
  1. 首先计算hash，遍历node数组，如果node是空的话，就通过CAS+自旋初始化
  2. 如果当前数组的位置是空，则直接通过cas自旋写入
  3. 如果hash==moved，说明需要扩容，执行扩容
  4. 如果都不满足，就是用synchronized写入数据，写入数据同样判断链表、红黑树，链表写入和hashmap一样，key hash一样就覆盖，反之就尾插法，链表唱过超过8就转为红黑树

get

通过计算key hash ,如果key hash相同就返回，如果是红黑树按照红黑树获取，都不是就遍历链表获取

volatile的原理

相比于synchronized通过加锁来解决共享变量的内存可见性问题，volatile是更轻量的选择，他没有上下文切换的额外开销成本。使用volatile声明的变量，可以确保被更新的时候对其他线程的立即可见，volatile使用内存屏障来保证指令不会发生指令重排

可见性

线程都是从主内存中读取共享变量到工作内存中来操作的，完成之后再把结果写回主内存，但是这样会带来可见性的问题，假如现在我们是两级缓存的双核cpu架构，包含L1、L2两级缓存

1.线程A首先获取变量x的值，由于最初两级缓存都是空，所以直接从主内存中读取x，假设x的初始值是0，线程A读取之后将x值修改为1，同时写回主内存

2.线程b也同样读取X的值，由于L2缓存已经有缓存X=1了，所以直接从L2缓存读取，之后线程b将x修改为2，同时写回L2和主内存

那么这个时候线程a在想修改x的值，因为L1缓存已经存在x=1了，那么线程a就会在这基础上操作，问题就产生了，线程b的修改对于线程a来说是无感知的

这个时候，如果x是用volatile修饰的话，当线程a再次读取x的话，cpu会根据缓存一致性协议强制线程a重新从主内容加载最新的值到自己的工作内存中，而不是直接使用缓存中的值

内存屏障

volatile修饰之后会加入不同的内存屏障保证可见性的问题能正确的执行

  1. StoreStore屏障，保证上面的普通写不会和volatile写发生重排序
  2. StoreLoad屏障，保证volatile写与后面可能的volatile读写不会发生重排序
  3. LoadLoad屏障，禁止volatile读与后面的普通读重排序
  4. LoadStore屏障，禁止volatile读与后面的普通写重排序

ThreadLocal原理

threadLocal可以理解为本地变量，他会在每个线程都创建一个副本，那么在线程之间访问内部副本变量就行了，做到了线程之间的相互隔离，相比于synchronized的做法是用空间换时间

theadLocal有一个静态内部类ThreadLocalMap,threadLocalMap又包含了一个entry数组，entry本身是一个弱引用，他的key是指向threadLocal的弱引用，entry具备了保存k v的能力

弱引用的目的是为了防止内存泄漏，如果是强引用，那么threadLocal对象除非线程结束，否则就无法被回收，弱引用则会在下一次gc的时候被回收

但是这样还是会存在内存泄漏的问题，加入key和threadLocal对象被回收后，entry中就存在key为null，但是value存在值的对象，但是永远无法访问到，除非线程结束运行

线程池原理

核心参数：

  1. 最大线程数
  2. 核心线程数
  3. 活跃时间
  4. 阻塞队列
  5. 拒绝策略

当提交一个新任务到线程池时，具体的执行流程：

  1. 当我们提交任务，线程池会根据核心线程数大小创建若干任务数量线程执行任务
  2. 当任务的数量超过核心线程数后，后续的任务将会进入到阻塞队列中阻塞排队
  3. 当阻塞队列也满的时候，那么将会判断是否达到最大线程数，如果没达到就会创建额外线程来执行任务，如果任务处理完成，最大线程数额外创建的线程则会等待活跃时间之后自动销毁
  4. 当最大线程数都满了，阻塞队列也是满的，那么就会根据不同的拒绝策略来处理
拒绝策略
  1. abortPolicy: 直接丢弃任务，抛出异常，这是默认策略
  2. callerRunPolicy：只用调用者所在的线程来处理任务
  3. discardOldestPolicy:丢弃等待队列中最旧的任务，并执行当前任务
  4. discardPolicy:直接丢弃当前任务，也不抛出异常
