当输入网址后，到网页显示，中间发生了什么？

发表于2021-05-11|更新于2021-05-11|面试
|阅读量:

HTTP

浏览器做的第一步工作就是解析url

从图中可知，url时间上是请求服务器里的文件资源

当没有路径名时，就代表访问根目录下事先设置的默认文件，也就是/index.html

对url进行解析之后，浏览器确定了web服务器和文件名，接下来就是根据这些信息生成http请求消息了

dns

通过浏览器解析url并生成http消息后，需要委托操作系统将消息发送给web服务器

在发送消息之前，需要确定web服务器的地址，即ip地址

有一种服务器专门保存web服务器域名和ip的对应关系，它就是dns服务器

dns中的域名都是用句点来分割的，比如www.server.com，这里句点代表了不同层次之间的界限

在域名中，越靠右的位置表示其层级越高

根域是在最顶层，它的下一层就是com顶级域，在下面就是server.com

  • 根dns服务器
  • 顶级域dns服务器（com）
  • 权威dns服务器（server.com）

根域的dns服务器信息保存在互联网中所有的dns服务器中

这样一来，任何dns服务器都可以找到并访问根域dns服务器

因此，客户端只要能够找到任意一台dns服务器，就可以通过他的根域dns服务器，然后在顺藤摸瓜最好找到位于下层的某台目标dns服务器

域名解析的工作流程
  1. 客户端首先会发出一个dns请求，问www.server.com的ip是啥，并发给本地的dns服务器（也就是客户端TCP/IP设置中填写的dns服务器地址）
  2. 本地域名服务器收到客户端请求后，如果缓存的表格里能够找到www.server.com，则他就直接返回ip地址，如果没有，本地dns会去问它的根域名服务器
  3. 根dns收到本地dns的请求后，发现后缀是.com，就告诉本地dns服务器：”www.server.com这个域名归.com区域管理",我给你.com顶级域名服务器地址
  4. 本地dns收到顶级域名服务器的地址后，继续问他，你知道www.server.com的ip地址么
  5. 顶级域名服务器说：“我给你负责www.server.com区域的权威dns服务器地址，你去问问他”
  6. 本地dns转向去问权威dns服务器，“server.com的权威dns服务器，它就是域名解析的出处”，于是将对应的ip告诉本地dns
  7. 本地dns再将ip地址返回给客户端，客户端和目标建立连接

在这个过程中，除了实际解析的权威dns服务器知道ip以外，其他的dns服务器都是扮演指路人的角色

协议栈

通过dns获取ip之后，就可以把http的传输工作交给操作系统的协议栈

协议栈的内部分为几个不同部门，分别承担不同的工作

应用程序（浏览器）是通过socket库来委托协议栈工作，协议栈的上半部有两块，分别是负责收发数据的tcp和udp协议，他们两会接受应用层的委托，执行收发数据的操作

协议栈的下面一半是用ip协议控制网络包收发操作，在互联网上传数据时，数据会被切分成一块块的网络包，而将网络包发送给对方的操作就是由ip负责的

此外，ip还包括了ICMP和ARP协议

  • ICMP用于告知网络包传送过程中产生的错误以及各种控制信息
  • ARP用户根据ip地址查询响应的以太网mac地址

ip下面的网卡驱动程序负责控制网卡硬件，而最下面的网卡负责完成实际的收发操作

tcp

http是基于tcp协议传输的

tcp包头格式

  • 源端口号和目标端口号是不可少的

  • 包的序号，这个是为了解决包乱序的问题

  • 确认序列是为了确认发出去对方是否收到，如果没收到就继续发，知道送达，这个是为了解决不丢包的问题

  • 状态位

    1. syn 发起一个连接
    2. ack 回复
    3. rst 重新连接
    4. fin 结束连接

    tcp是面向连接的，因而双方要共同维护连接的状态

  • 窗口大小，tcp是要做流量控制的，通信双方各声明一个窗口（缓存大小），标识自己当前能够处理能力

三次握手

tcp在传输数据之前，要先三次握手建立连接

这个所谓的链接，只是双方计算器里维护一个状态机

  • 开始阶段，客户端和服务端都处于closed状态，先是服务端主动监听某个端口，处于listen状态
  • 然后客户端主动发起连接syn,之后处于syn_sent状态
  • 服务端收到发起的连接，返回syn，并且ack客户端的syn,之后处于syn_rcvd状态
  • 客户端收到服务端发送的syn和ack之后，发送ack的ack之后处于established状态，因为它一收一发成功了
  • 服务端收到ack的ack之后，处于established状态，因为它也一收一发成功了

所以tcp的三次握手的目的是保证双方都有发送和接受的能力

ip

tcp模块在执行连接、收发、断开等各阶段操作时，都需要委托ip模块将数据封装成网络包发送给通宵对象

ip包头格式

在ip协议里面需要有源地址ip和目标地址ip

因为http是经过tcp传输的，所以在ip包的协议号，要填写06（十六进制）,表示协议为tcp

最后形成的网络包报文

MAC

生成ip头部之后，接下来网络包还需要在ip头部的前面加上mac头部

mac头部是以太网使用的头部，他包含了接收方和发送方的mac地址等信息

一般在tcp/ip通信里，mac包头的协议类型只是用

  • 0800：ip协议
  • 0806：ARP协议

mac的发送方和接收方是如何确认的？

发送方的mac地址获取比较简单，mac地址是在网卡生产时就写入ROM中的，只要将这个值读取出来，然后写入到mac头部就可以了

接收方 mac地址就相对比较复杂了

现在我们已经通过dns知道了接收方的ip地址了，那么通过ip怎么才能获取到mac地址呢？

这个时候就需要通过ARP协议了

ARP协议会在以太网中已广播的形式，对以太网所有的设备喊出：”这个ip地址是谁的？请把你的mac地址告诉我”

如果对方和自己处于一个子网中，那么通过上面的操作就可以得到对方的mac地址

网卡

ip生成的网络包只是存放在内存中的一串二进制数字信息，没有办法直接发送给对方，因此，我们需要将数字信息转换为电信号

负责执行这一操作的就是网卡，要控制网卡还需要靠网卡驱动程序

网卡驱动从ip模块获取到包之后，会将其复制到网卡内的缓存区中，接着会在其开头加上报头和起始帧分解符，在末尾加上用于检测错误的帧校验序列

  • 起始帧分解符是一个用来表示包起始位置的标记
  • 末尾的fcs(帧校验序列)是用来检查包传输过程中是否有损坏

最后网卡会将包转为电信号，通过网线发送出去

交换机

交换机的设计是将网络包原样转发到目的地。交换机工作在mac层，也称为二层网络设备

首先，电信号到达网线接口，交换机里的模块进行接收，接下来交换机里的模块将电信号转换为数字信号，然后通过末尾的fcs检验错误，如果，没问题就放到缓存区

计算机网卡本身具有MAC地址，并通过核对收到的包的接收方的MAC地址判断是否发送给自己的，如果不是发给自己的就丢弃；相对的，交换机的端口不核对接收方mac地址，而是直接接受所有的包并放入缓存区中

交换机根据mac地址表查找mac地址，然后将信号发送到对应的端口

当mac地址表找不到指定的mac地址会怎么样？

地址表中找不到指定的mac地址，这可能是因为具有该地址的设备还没有向交换机发送过包，或者这个设备一段时间没有工作，导致地址被删除了

这种情况下，交换机无法判断应该把包发给那个端口，只能将包转发给除了源端口之外的所有端口上，无论该设备连接到哪个端口上都能接收到这个包

以太网的设计本来就是将包发送到整个网络的，然后只有相应的接收者才能接收包，其他设备则会忽略这个包

路由器

网络包经过交换机之后，到达了路由器，并在此被转发到下一个路由器或目标设备

这一步转发的原理和交换机类型，也是通过查表判断包转发的目标

不过具体还是有区别的

  • 路由器是基于ip设计的，俗称三层网络设备，路由器的各个端口都具有mac地址和ip地址
  • 交换机是基于以太网设计的，俗称二层网络设备，交换机的端口不具有mac地址

服务器与客户端

数据包抵达了了服务器，服务器开始解析数据包

服务器会先解析mac头部，查看是否和服务器自己的mac地址符合，符合就将包收下

接着解析数据包的ip头，发现ip地址符合，根据ip头中的协议项，知道自己上层是tcp协议

然后开始解析tcp头，根据tcp头部的序列号，查看这个序列包是不是我想要的，如果是就放到缓存中，然后返回一个ack，如果不是就丢弃。tcp头部里面还有端口号，http的服务器正在监听这个端口号

这个时候服务器就知道这个是http需要的包，就把这个包发给http进程

http进程看到，原理这是要请求一个页面，于是就把这个网页封装到http响应报文里

http响应报文同样需要加上tcp、ip、mac头部，不过这次源地址是服务器，目标地址是客户端

经过网卡，将数据交给交换机，在转给路由器，路由器把数据发给目标客户端的路由器

同样的，客户端也开始一层层解析，最终得到http响应报文，交给浏览器去渲染页面，

最后，客户端离开，向服务器发起tcp四次挥手，至此双方的链接就断开了

总结

  • 当我们输入网址的时候，首先浏览器会解析url，通过url可知发送的协议http,以及服务器域名和服务器文件资源位置
  • 通过dns解析获取该域名的ip
  • 通过tcp协议，浏览器通过三次握手和服务器建立连接
  • 添加ip头报文
  • 添加mac头报文
  • 通过网卡将数据信号转换为电信号
  • 通过交换机将电信号转换为数字信号，同时转发给路由器
  • 路由器将广播找到目标mac的路由器，将数据发送过去
  • 服务接收到数据包，开始解析报文头，根据ip报文头部，知道当前数据是tcp协议，然后找到对应的端口，将这个数据包转给当前端口的http进程
  • 获取html页面输出到http响应报文数据中，然后一层层添加tcp、ip、mac头部，返回给浏览器
  • 浏览器接收到响应后，解析得到响应报文，同时返回一个ack给服务器，然后浏览器渲染数据
