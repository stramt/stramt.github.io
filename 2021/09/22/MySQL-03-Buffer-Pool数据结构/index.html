MySQL-03(Buffer Pool数据结构)

发表于2021-09-22|更新于2021-09-28|MySQL
|阅读量:

1.如何设置Buffer Pool的大小

buffer pool本质上还是内存组件，因此就是有上限的，默认情况下是128mb,实际生产环境中，还是有点偏小了，因此我们可以调整它的大小。

比如再16核32g的机器上，我们完全可以将Buffer Pool设置为2G

2.数据页

假设现在我们的数据库中有一块区域是Buffer Pool了，那么我们的数据是如何放在Buffer Pool中的？

我们都知道数据库的数据模型就是表+字段+行的概念，也就是说数据库中是存在一个一个的表，每个表有多个字段，然后一个表中存在多行数据，每行数据都有自己的字段值，难道我们的数据是一行一行的放在buffer pool中的吗？

很显然，不是的，这样效率太低了。

mysql在这里引入了数据页的概念

他把很多行数据放在了一个数据页中，每次加载到buffer pool都按照页来加载

所以实际上假设我们要更新一行数据，此时数据库会找到这行数据所在的数据页，然后从磁盘文件里把这行数据所在的数据页直接给加载到Buffer Pool中

3.磁盘上的数据页和Buffer Pool中的缓存页是如何对应起来的？

实际上，磁盘中存放的数据页默认大小为16kb,也就是说每次从磁盘中加载的大小是16kb的内容

默认情况下，Buffer Pool中的数据页我们称之为缓存页，缓存页的大小是和数据页大小一一对应的，都是16kb

如图，假设我们给Buffer Pool设置128mb的缓存，那么在buffer pool中我们就可以缓存8192页的内容了

4.缓存页对应的描述信息是什么？

实际上在buffer pool中的缓存页，每个缓存页都会存在一个描述信息，这个描述信息主要是用来记录这个缓存页的基本信息的。

比如，这个是数据页所属的表空间、数据页的编号、这个缓存页在buffer pool中的地址和其他一些信息，这些描述信息本身也是一块数据，它会放在缓存页的最前面，相当于头部

如图所示，所有的缓存页的描述信息都放在一块区域，然后通过指针指向对应的缓存页

而且，我们要注意一点，buffer pool中的描述信息大概相当于缓存页大小的5%左右，也就是每个描述信息大概在800个字节左右的大小，然后假设你设置的buffer pool大小是128mb,实际上buffer pool的真正大小会超出一些，可能有130mb的样子，这就是因为它里面还要存放每个缓存页的描述数据

5.初始化Buffer Pool

数据库只要一启动，就会按照你设置的Buffer Pool大小，稍微再加大一点，去找操作系统申请一块内存区域，作为Buffer Pool的内存区域

然后当内存区域申请完毕之后，数据库就会按照默认的缓存页的16kb的大小以及对应的800个字节左右的元数据的大小，再buffer pool中划分一个个缓存页和一个个对应的元数据

只不过，这个时候，buffer pool中的一个个缓存页都是空的，里面什么都没有，要等数据库运行起来之后，当我们需要对数据执行增删改查的时候，才会把数据对应的页从磁盘文件读取出了，放到buffer pool的缓存中

当数据库运行起来之后，肯定是会不停的增删改查，此时就需要不停的从磁盘上读取一个个数据页放到buffer pool中对应的缓存页里去，把数据缓存起来

但是此时会有一个问题，从磁盘上读取数据页放到buffer pool中的缓存页的时候，那些缓存页是空闲的？

因为再默认的情况下，磁盘上的数据页和缓存页是一一对应起来的，都是16kb

所以，我们必须要要知道buffer pool中哪些缓存页是空闲的状态。

free链表

数据库会为buffer pool设计一个free链表，他是一个双向链表数据结构，再这个free链表里，每个节点就是一个空闲的缓存页的元数据的地址，也就是说，只要你一个缓存页是空闲的，那么他的元数据块就会被放到这个free链表中

刚开始数据库启动的时候，可能所有的缓存页都是空闲的，因为此时可能是一个空的数据库，一条数据都没有，那么此时所有的缓存页的元数据都会被放到这个free链表中

如图可知，再free链表中会有一个基础节点，他会引用链表的头节点和尾节点，里面还存储了链表中有多少个元数据的节点，也就是有多少个空闲的缓存页

free链表本身是buffer pool中的元数据组成的，每个元数据都是存在两个指针，一个free_pre，一个free_next，分别指向自己的上一个free链表的结点，以及下一个free链表的结点

也就是说free链表中的元数据本身就是buffer pool中既有的元数据，并不是将这些元数据复制一份再放到链表中的，因此free链表所占用的空间只有基础节点这个节点，大小再40个字节。

那么，数据库是如何将磁盘上的数据页读取到buffer pool中的缓存页？

首先，我们需要从free链表中获取一个元数据，然后找到对应的这个元数据的空闲的缓存页，接着我们就可以将磁盘上的数据读取到对应的缓存页中，同时把相关的一些元数据写入到缓存页的元数据中去，最后将这个元数据从free链表中去除即可

如何知道数据页有没有被缓存

到这里，我们就知道，当执行增删改查的时候，首先需要查询这个数据页有没有被缓存

如果没有被缓存，就走上面的逻辑，从free链表中找到一个空闲的缓存页，从磁盘上读取数据页写入缓存页，写入元数据，然后再从free链表中移除这个元数据

但是如果这个数据页已经存在了，那么就可以直接使用

因此，数据库还会有一个hash表数据结构，他会用表空间号+数据页号，作为一个key，然后缓存页的地址作为value.

当你要使用一个数据页的时候，通过”表空间号+数据页号”作为key去这个表里查一下，如果没有，就读取数据页，如果存在，就说明已经被缓存了

6.Buffer Pool碎片化

因为BufferPool的大小是我们自定义的，所以会出现当buffer pool划分完所有的缓存页和元数据之后，还剩余内存，这些内存又不足以放下任何一个缓存页，所以就产生了内存碎片。

因此，数据库在buffer pool中划分缓存页的时候，会让所有的缓存页和元数据都紧密的挨在一起，这样尽可能的减少内存碎片的产生

7.脏数据页到底为什么会脏

当我们执行增删改查的时候，如果发现数据页没缓存，那么就会基于free链表找到一个空闲的缓存页，然后读取到缓存页里去，但是如果已经缓存了，那么下一次就必然会直接使用缓存页

反正不管怎么样，你要更新的数据页都会在buffer pool的缓存页里面，增删改查都是在内存操作的

当更新了buffer pool中缓存页的数据，就会导致缓存页和数据页的数据不一致，就会产生脏数据、脏页

但是在内存中更新的脏页的数据，都是要刷新回磁盘文件的，但是会存在一个问题，并不是所有的缓存页都需要刷回磁盘的，因为有些缓存页并没有被修改过

因此，数据库在这里引入了另外一个链表，flush链表，这个链表的本质也是通过缓存页的元数据中的两个指针，让被修改过的缓存页的元数据，组成一个双向链表

8.Buffer Pool的缓存页不够用了

当我们执行增删改查操作的时候，此时Buffer Pool的缓存页已经被占满了，而我们需要查询的数据又不在缓存页中，我们需要将磁盘上的数据页读取到Buffer Pool中执行操作，这时候该怎么办？

这个时候一般情况下，我们会淘汰一些缓存页数据，那么如何淘汰呢？

我们淘汰缓存页数据是为了释放buffer pool，这个时候我们就需要将这个缓存页的数据flush到磁盘上的数据页上，然后再清空这个缓存页

那么，我们需要淘汰哪些缓存页呢？

缓存命中率

假设我们现在有两个缓存页，一个缓存页的数据，经常会被修改和查询，比如100次请求中，有30次都在查询和修改这个缓存页的数据

另一个缓存页，100次请求中可能就只有一次修改和查询

这两个缓存页，第一个缓存命中率就比较高，第二个就比较低

LRU链表来判断缓存页是否常用

通过缓存命中率我们能够知道那些缓存页常用，哪些不常用

因此，在这里数据库通过引入LRU链表来选择最少使用的缓存页

假设我们从磁盘上加载一个数据页到缓存页的时候，就会把这个缓存页的元数据放到LRU链表的头部去，因此，只要存在缓存页，他就会再LRU中，当我们查询或修改一个缓存页的时候，同时也会将这个缓存页从LRU链表中的某个位置，再放到LRU头部中，这样就可以保证，越接近头部的缓存页，访问频率越高，越接近尾部的缓存页，访问频率越低

9.多线程访问Buffer Pool

我们知道Buffer Pool其实就是有缓存页+元数据+free+flush+lru组成的，这些都是共享的，因此再多线程访问下，会出现并发的问题

一般情况下，我们遇到并发的时候，都会通过加锁来解决，但是这里思考一下，如果加锁，数据库的性能是不是就会变的很差？

其实也不是，因为buffer pool本身就是一块内存区域，哪怕是所有的操作都加锁串行化也不会太影响数据库的性能

但是，还有一个问题，如果都是再内存中操作到还好，但是当我们再缓存页中没有查询到数据的时候，需要从磁盘上的数据页加载到缓存页的时候，这时候是会进行一次磁盘IO操作的，IO操作是很消耗资源的，如果是加锁串行化的话，发生大量的磁盘IO操作，就会导致性能低下

多个Buffer Pool优化并发能力

一般情况下，MySQL默认的规则是，如果你给Buffer Pool分配的内存小于1GB，那么最多只会给你一个Buffer Pool

但是如果你的内存够大，那么你必然就会给Buffer Pool分配较大的内存，比如给8G内存，那么我们就可以设置多个Buffer Pool

[server]

innodb_buffer_pool_size = 8589934592

innodb_buffer_pool_instances = 4

这样，我们就设置了4个buffer pool，每个buffer pool的内存为2g

这个时候再有多线程访问的时候，就会能将压力平摊到每个buffer pool中

生产环境中buffer pool设置多少内存

一般设置为机器内存的50%~60%

buffer pool的总大小=（chunk大小buffer pool数量）2

默认情况下，chunk大小是128m,假设此时机器内存是32g,那么我给buffer pool的大小在20g左右

假设我设定的buffer pool的数量为20个，那么chunk大小*buffer pool数量=16 *128=2048m,然后buffer pool的总大小为20g,就符合规则

同样的，比如我将buffer pool的数量设置为32个，那么就是5倍，也符合规则

show engine innodb status

通过指令查询具体参数

  1. total memory allcated buffer pool的总大小
  2. buufer pool size buffer pool一共能容纳多少缓存页
  3. free buffers 这就说free链表中一共有多少空闲的缓存页可用
  4. Database pages 和Old database pages，就是说lru链表中一共有多少个缓存页，以及冷数据区域缓存页的数量
  5. Modified db pages flush链表中的缓存页数量
  6. Pending writes、Pending reads，即将从flush链表刷入磁盘的数量，等待从磁盘加载到缓存页的数量
  7. Pages made young和not young 这是说已经lru冷数据区域里访问之后转移到热数据区域的缓存页数量，已经在lru冷数据区域里1s内被访问了没进入热数据区域的缓存页的数量
  8. youngs/s和not youngs/s,这就是说每秒从冷数据区域进入热数据区域的缓存页的数量，以及每秒在冷数据区域里被访问了但是不能进入热数据区域的缓存页的数量
  9. LRU len lru链表里的缓存页的数量
  10. I/O sum:最近50s读取磁盘页的总数
  11. I/O cur 现在正在读取磁盘页的数量
